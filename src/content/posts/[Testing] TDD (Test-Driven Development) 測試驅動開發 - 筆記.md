## 核心觀念  
TDD 是一種「開發流程與節奏」。傳統開發是先寫程式碼再補測試，而 TDD 則是反過來——**在還沒寫任何實際功能（實作碼）之前，先寫測試程式碼！** 透過測試來引導、架構你的程式設計。

## TDD 的核心循環：紅燈、綠燈、重構 (Red-Green-Refactor)

TDD 的開發節奏非常規律，通常被稱為 **「3R 循環」**：

### 🔴 階段一：紅燈 (Red) —— 先寫一個必定失敗的測試
* **核心任務**：在還沒有實作功能前，根據需求先想好「這個功能做完後應該長怎樣」，然後直接寫測試。因為功能還不存在，執行測試時**絕對會失敗（亮紅燈）**。
* **💡 為什麼**：確保你的測試真的有抓到「功能還沒做好」的狀態，避免寫出一個不論實作怎麼改都會盲目通過的無效測試。

### 🟢 階段二：綠燈 (Green) —— 用最快、最髒的方式讓測試通過
* **核心任務**：寫出「剛好能讓測試通過」的實作程式碼。在這個階段，**不需要追求程式碼的美感、效能、完美架構**。不擇手段，只要能讓測試從紅燈變成**綠燈（Passed）**就好。
* **💡 為什麼**： **讓你一次只專注在一件事上（滿足核心需求），避免提早過度設計（Over-engineering）。**

### 🔵 階段三：重構 (Refactor) —— 整理與優化程式碼
* **核心任務**：當測試亮起安全的綠燈後，在**測試防護網的保護下**，安心地整理、優化剛剛寫得很髒的程式碼。你可以優化變數命名、抽離重複邏輯、提升效能、調整架構。
* **💡 為什麼**：每改一步就跑一次測試，只要測試還是綠燈，就代表你**沒有改壞任何既有功能**！



## 實戰模擬：以「迷因產生器」為例

假設 PM 新增需求：*「當輸入框留白時，畫面上必須即時顯示提示字『欄位不能留白』」*。

1. **🔴 Red (紅燈)**：功能還沒寫。直接寫測試模擬使用者清空輸入框，並斷言 `screen.getByText("欄位不能留白")` 應存在。執行測試 ➡️ **失敗（紅燈）**。  
2. **🟢 Green (綠燈)**：立刻去 `Main.jsx` 裡，用最直覺的 `if (value === "")` 將狀態塞入 `error`。執行測試 ➡️ **成功（綠燈）**。  
3. **🔵 Refactor (重構)**：發現這個防呆驗證之後很多地方會用到，於是在測試保護下，將它抽離成獨立的 `validateInput()` 外部工具函式。重構完再跑測試 ➡️ **依然綠燈**。完美收工！



## TDD 的優缺點

| 優點 | 缺點 / 挑戰 |  
| :--- | :--- |  
| **無與倫比的信心**：有測試當護身符，未來進行大幅度重構或升級套件時完全不害怕。 | **思維切換困難**：習慣先寫 code 的大腦需要重新適應「先想規格、先寫測試」的反直覺流暢度。 |  
| **規格即測試**：測試檔案就是專案最完美的「活體說明書」，新進工程師看測試就知道業務邏輯。 | **前期速度變慢**：開發初期的速度會比直接盲寫 code 還慢，需要投入較多前期時間成本。 |  
| **代碼設計品質高**：為了解耦好測試，你會被迫寫出職責單一、高內聚、好維護的漂亮元件。 | **規格頻繁變動**：若專案初期方向極度不穩定、天天改規格，測試碼也必須跟著頻繁修改，維護成本高。 |

---

## 總結

**"Write tests first. Make it run, make it right, make it fast."**  
（先寫測試。先讓它能跑、再讓它跑得對、最後讓它跑得快。）
